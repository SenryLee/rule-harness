# 规则抽取工作流 —— 导入与使用说明

配套文件：`dify-workflow-规则抽取.yml`（可直接导入 Dify 1.13.3）

---

## 这个工作流做什么

```
开始(上传多文件) → 提取文档文本 → 调用规则梳理后端(同步抽取) → 解析规则JSON → 结束(输出 rules)
```

5 个节点，一条直线，**没有循环**。最终输出两个字段：`rules`（规则数组）、`total`（规则条数）。

### 为什么是这个设计（重要）

抽取要几分钟，而 Dify 1.13.3 没有"循环等待"节点，所以做了两件事绕开它：

1. **后端加了一个"一次搞定"接口** `POST /api/dify/extract`：它在一次请求里跑完抽取再返回，Dify 不用自己轮询。
2. **文件交给 Dify 自带的"文档提取器"先转成文本**再发给后端：这样绕开了 Dify HTTP 节点上传文件 ~800KB 的大小限制，PDF/Word 多大都行。

> 代价：文本模式只走"正文类"抽取，**不包含 Word 批注/修订那两条管道**。如果你要保留批注/修订，见文末「进阶」。先用这个版本跑通即可。

## 第一步：先部署新后端接口（必做）

这个工作流依赖后端新接口 `/api/dify/extract`，它现在还没上线。在你电脑终端粘贴这行推送部署：

```bash

git add backend/routes/dify_routes.py "docs/dify-workflow-规则抽取.yml" "docs/dify-workflow-导入与使用说明.md" && \
git commit -m "feat(dify): 新增同步抽取接口 /api/dify/extract + 规则抽取工作流" && \
git push origin main
```

等 Render 部署完（一两分钟），验证接口已上线（返回需要 files/text 的提示就说明通了）：

```bash
curl -X POST https://rule-harness-demo.onrender.com/api/dify/extract
```

---

## 第二步：导入工作流

1. Dify → **工作室（Studio）** → **导入 DSL / Import**（创建应用处选"导入 DSL 文件"）。
2. 选择 `dify-workflow-规则抽取.yml`。
3. 导入后会看到 5 个节点连成一条线。

---

## 第三步：试运行

1. 右上角 **运行 / Run**。
2. 在 `上传文件（可多选）` 处上传 1 份小一点的合同（先别传几百页的）。
3. 等待——HTTP 节点会一直转到后端抽完（已设最长等待 300 秒）。
4. 跑完看 **结束** 节点输出：`rules` 是规则数组，`total` 是条数。全绿就成了。

---

## 常见问题

- **HTTP 节点超时/报错**：文档太大、抽取超过 300 秒会超时。先用小文件；要处理大文件就拆开分次传，或告诉我帮你把超时调更高 / 改成异步轮询版。
- **rules 为空**：可能文本没提取出来（扫描件 PDF 是图片，要先 OCR），或文件内容里确实没有可抽取的规则。先用一份文字版合同测试。
- **想要可读摘要而不是 JSON**：在"解析规则JSON"后面加一个 LLM 节点，把 `{{#解析规则JSON.rules#}}` 丢给它总结即可（需要的话我帮你加）。

## 进阶：保留 Word 批注/修订（用你已创建的工具，文件原样上传）

如果要走完整保真（含批注、修订等全部管道），就别用"文档提取器+文本"，改成**直接把文件传给后端**：

1. 把你已建的「规则梳理工具」**重新导入一次 schema**（用 `https://rule-harness-demo.onrender.com/api/dify/openapi.json`），它会多出一个 `extractSync` 动作（对应 `/api/dify/extract`，支持直接传文件）。
2. 在工作流里把中间两个节点（文档提取器 + HTTP 请求）换成**一个工具节点**，选 `规则梳理工具 / extractSync`，把 `files` 参数指到开始节点的上传文件。
3. 其余（解析 + 结束）不变。

> 这条路文件保真度最高，但 Dify 工具节点对超大文件也可能有上限；而且工具节点的参数要在画布上手动连，比导入 YML 麻烦些。建议先用上面的文本版跑通，再按需升级。

需要我直接给你出"保真版"的工作流，或加一个 LLM 摘要节点，告诉我即可。
