# Dify 工作流 + 自建服务 接入方法论（实施方案）

> 一句话概括这套打法：**胖后端、瘦工作流**——把复杂、严谨、性能敏感的能力写成代码部署成接口，Dify 工作流只负责把这些能力按业务 SOP「编排」起来。既保留工作流的严谨与可视化，又用代码补上 Dify 表达力和灵活性的短板。
>
> 本文是从「规则梳理工具」接入 Dify 的实战中沉淀出的**可复用方法论**，供你以后做更多复杂功能的 Dify 工作流时照着套。

---

## 一、核心理念

把一个复杂业务拆成两层，各干各擅长的事：

- **能力层（你的网站/后端）**：承载"重"的、复杂的、需要严谨保证的功能——文件解析、多管道抽取、置信度评分、数据持久化、和外部 API 的复杂交互等。用代码实现，意味着**可测试、可版本管理、可复用、性能可控**，不受 Dify 节点表达力的限制。
- **编排层（Dify 工作流）**：负责把能力层的接口、LLM 节点、判断分支按业务流程**串起来**。它的价值是**流程可视化、易调整、非技术同事也能看懂甚至微调**，天然适合做"分类→路由→调用→汇总"这种编排。

两层之间用 **HTTP 接口**连接。能力层对 Dify 而言就是一个"工具"，Dify 对能力层而言就是一个"调用方"。

---

## 二、什么该放代码、什么该放 Dify（决策准则）

下次拆一个新功能时，先按这张表分流：

| 特征 | 放哪里 |
|------|--------|
| 逻辑复杂 / 需要单元测试 / 要复用 | 代码（后端接口） |
| 性能敏感 / 大数据处理 / 文件解析 | 代码 |
| 要严格保证正确性（数值、合规） | 代码 |
| 流程会经常调整 / 要给人看懂 | Dify 编排 |
| 纯 LLM 判断、分类、总结、改写 | Dify 的 LLM 节点 |
| 多个能力的串联、条件分支、人工节点 | Dify 编排 |

**经验法则**：凡是"用 Dify 节点拼起来会很别扭、很多 code 节点、很难维护"的部分，就抽成一个后端接口；凡是"流程性的、需要灵活调整的"部分，留在 Dify。

## 三、整体架构

```
┌─────────────────────────────────────────────┐
│                Dify 工作流（编排层）            │
│   开始 → 预处理 → [调用能力接口] → LLM/判断 → 汇总 → 结束 │
└───────────────┬─────────────────────────────┘
                │ HTTP（JSON / form-data）
┌───────────────▼─────────────────────────────┐
│            你的网站 / 后端（能力层）            │
│  接口层  →  业务引擎（解析/抽取/评分/...）       │
│            └─ 数据持久化（SQLite/DB）           │
│  同一个后端同时还是「任务后台」：可视化、可管理     │
└─────────────────────────────────────────────┘
                │ 仅"重活"外包
                ▼  外部 LLM API（DeepSeek/通义等）
```

要点：**算力大头（LLM）外包给第三方 API**，所以能力层服务器本身很轻。能力层后端同时兼任"任务管理后台"——所有 Dify 发起的任务都能在网页端看到、查进度、看结果、管理。

## 四、实施步骤（可复制的标准流程）

### 步骤 1　拆解 SOP，划分两层
把业务流程写成自然语言 SOP，逐步标注：哪一步是"编排/判断"（留 Dify），哪一步是"重能力"（抽成接口）。产出一张节点拓扑草图 + 一份"要新增哪些后端接口"的清单。

### 步骤 2　设计并实现后端接口（接口契约先行）
为每个能力定义清晰的**接口契约**：路径、方法、输入（JSON 或 form-data）、输出（统一 JSON 结构）。建议统一返回结构，便于 Dify 侧解析：

```json
{ "id": "批次/任务ID", "status": "success|partial|running|failed",
  "total": 12, "data": [ ... ], "summary": { ... }, "error": null }
```

接口实现尽量**复用已有业务引擎**，只在外面包一层"接收输入→调引擎→返回 JSON"。

### 步骤 3　解决"异步与超时"（这套打法最关键的一步）
Dify 的工具/HTTP 调用有大约 **1 分钟超时**，而复杂能力常常要跑几分钟。两种标准解法，按任务时长二选一：

- **同步阻塞接口**（任务能在限时内完成，或后端所在平台允许长请求）：一次请求里跑完再返回结果，Dify 一个节点搞定，最简单。把 HTTP 节点读超时调高。
- **异步两段式**（任务很长）：上传接口立即返回 `id`；再提供"查状态"和"取结果"两个接口。由于 Dify 1.13.3 **没有 loop/wait 节点**，不要在画布里做轮询循环——而是让**网页后台**承担监控（见步骤 5），或把"等待"放进服务端。

### 步骤 4　在 Dify 接入（节点选型有讲究）
- **调用能力接口：优先用 HTTP 请求节点**直连后端。它自包含、好导入、不依赖你账号里具体工具的内部 ID，也避开了"自定义工具在工作流里绑不住文件变量"的坑。
- **读类查询接口（GET、只传字符串）**：适合做成**自定义工具**（OpenAPI 一键导入），也适合给 Agent/Chatflow 用。
- **文件输入**：别直接把二进制怼进 HTTP 节点（有 ~800KB 上限且 schema 易错）。用 **Dify 自带的「文档提取器」先把文件转成文本**再发后端；要保留文件原貌（如 Word 批注/修订）时，才走"文件直传后端"的接口。

### 步骤 5　把任务接入"网页后台"（可观测 + 可管理）
让后端接口在受理 Dify 任务时，把任务**登记进和网页端共享的任务列表/进度状态**。这样 Dify 一发起，网页端立刻可见、能实时看进度、看结果、删除、二次处理——网页版就成了 Dify 的统一后台。（前提：后端单进程/共享状态；多实例时要落库。）

### 步骤 6　测试与上线
1. **先脱离 Dify 验后端**：用 curl / 自动化测试把每个接口跑通（输入→输出 JSON 正确）。
2. **再接 Dify**：导入/搭好工作流，用小样本端到端跑通，确认 `id` 贯通、状态正确、结果可解析。
3. 调优超时、并发、错误分支。

## 五、踩过的坑 → 通用对策（重要参考）

| Dify 约束 | 对策 |
|-----------|------|
| 工具/HTTP 节点约 1 分钟超时 | 同步阻塞接口（限时内）或异步两段式（长任务） |
| 1.13.3 无 loop / wait 节点 | 不在画布做轮询；改服务端阻塞，或异步 + 网页端监控 |
| HTTP 节点文件上传 ~800KB、文件体 schema 易错 | 用「文档提取器」转文本再传；大文件保真走后端直传接口 |
| 自定义工具在工作流里难绑文件变量、需真实 provider_id | 工作流里用 HTTP 节点直连后端；自定义工具留给读类 GET / Agent |
| Dify 侧调试信息有限 | 统一返回结构 + 后端结构化日志，排错看后端日志 |

## 六、优势

- **严谨 + 灵活兼得**：核心逻辑由代码保证正确性与性能，业务流程由 Dify 可视化编排、随时调整。
- **能力可复用**：同一个后端接口可被多个工作流、甚至其他系统调用；换掉 Dify 也不影响核心能力。
- **可维护、可演进**：复杂逻辑在代码里有测试、有版本、能回滚；不用把一堆逻辑塞进难维护的 code 节点。
- **协作友好**：工作流让非技术同事也能看懂业务流程、做轻量微调，分工清晰。
- **统一后台**：网页端兼任任务管理台，所有 Dify 任务可见可管。

## 七、劣势 / 风险

- **两套系统，运维复杂度上升**：部署、版本、联调都要管两边。
- **接口契约是耦合点**：改了接口的输入/输出，对应工作流要同步改，否则断链。
- **异步与超时是反复出现的坑**：每个长任务都要单独设计"怎么等"。
- **状态/数据一致性**：内存态、单实例是临时方案；要稳就得落库、加幂等。
- **安全暴露面**：能力接口一旦公网可达，需要鉴权，否则有被滥用/刷量风险。
- **可观测性弱**：Dify 画布里看不到后端内部，问题定位依赖后端日志。

---

## 八、后续补充建议（让这套打法成为可复制的"生产线"）

1. **沉淀脚手架**：把"统一接口返回结构"+"一个标准 HTTP 节点配置模板"+"鲁棒解析 JSON 的 code 节点"固化下来，新功能照抄，少踩坑。
2. **后端工程化**：接口加 **API Key 鉴权**；统一错误返回结构；关键接口做**重试与幂等**；上**结构化日志**和 `/health` 健康检查；任务状态**落库**（替代内存态，避免重启丢历史、支持多实例）。
3. **长任务标准化**：把"异步两段式 + 网页端监控"定为长任务的默认范式；大文件统一走后端直传接口。
4. **工作流治理**：工作流命名/版本规范，定期**导出 DSL 备份**；把"轮询""解析 JSON""调用能力接口"等通用片段做成可复用的子流程或模板。
5. **测试前移**：后端接口先 `curl`/`pytest` 跑通再接 Dify；端到端用固定小样本做回归，改接口后先回归再上线。
6. **监控告警**：给后端加基础监控（接口耗时、失败率），长任务失败能主动发现。
