# 规则梳理工具接入 Dify —— 小白逐步操作手册

> 你的后端已上线且可用：`https://rule-harness-demo.onrender.com`
> 目标：在 Dify 工作流里上传一份法律文件 → 自动抽出规则。

---

## 先理解 3 件事（30 秒）

1. 抽取很慢（要几分钟），而 Dify 的工具调用大约 1 分钟就超时。所以**不能"上传并当场等结果"**，必须分三步：
   - **第①步 上传**：把文件传过去，马上拿到一个 `batch_id`（像取件码）。
   - **第②步 查状态**：拿着 `batch_id` 反复问"好了没"，直到 `status` 变成 `success`。
   - **第③步 取规则**：好了之后用 `batch_id` 把规则 JSON 取回来。
2. 这 3 步对应后端 3 个接口，正好可以一次性导入成 1 个自定义工具（含 3 个 Action）。
3. 唯一的坑：**在工作流里，自定义工具的"文件参数"有时绑不上文件变量**。所以"上传"这一步我给你两种做法，先试简单的，不行就用稳的（见第三部分）。

---

## 第一部分：创建自定义工具（二选一）

### 方式 A —— 一键 URL 导入（推荐，但要先重新部署一次后端）

我已经给后端加了一个"工具说明书"接口。你把代码推到 GitHub、Render 自动部署完成后，按下面做：

1. 打开 Dify → 顶部 **工具** → 左上 **自定义** → **创建自定义工具**。
2. 在 **Schema** 区域右上角点 **「+ 从 URL 中导入」**。
3. 粘贴这个地址，确定：

   ```
   https://rule-harness-demo.onrender.com/api/dify/openapi.json
   ```
4. 下方"可用工具"会自动出现 3 个 Action：`uploadFiles` / `getBatchStatus` / `getBatchRules`。

> 注意：这个 URL 在你**重新部署之后**才会生效（现在访问会 404）。还没部署就先用方式 B。

### 方式 B —— 直接粘贴 Schema（现在就能用，无需重新部署）

后端那 3 个接口现在已经在线，所以不用等部署，直接粘贴即可：

1. Dify → **工具 → 自定义 → 创建自定义工具**。
2. 把项目里 `docs/dify-openapi.yaml` 文件的**全部内容**复制，粘贴到 **Schema** 输入框。
3. 下方"可用工具"出现 3 个 Action 就说明成功了。

### 两种方式都要填的其它字段

| 字段 | 填什么 |
|------|--------|
| **名称** | 随意，例如 `规则梳理` |
| **Schema** | 见上（URL 导入 或 粘贴 yaml） |
| **鉴权方法** | 选 **「无 / None」**（后端目前没设密钥；以后加了再改成"请求头"） |
| 标签 / 隐私协议 / 免责声明 | 留空即可 |

点 **保存**。到这里"工具"就建好了。

## 第二部分：在工作流里用起来（核心）

进入 **工作室 → 新建 → 工作流**（Workflow），按下面搭节点。

### 节点 1：开始（Start）

- 添加一个输入字段，类型选 **文件 / File**（或"文件列表 File List"），变量名例如 `doc`。
- 这样用户运行时就能上传文件，文件会进入系统变量 `sys.files`。

### 节点 2：上传文件，拿 batch_id

先试**简单做法**，不行再换**稳的做法**：

**简单做法（用刚建的工具）**：加一个 **工具节点**，选 `规则梳理 / uploadFiles`。
- 如果它的 `files` 参数能让你选到 `sys.files`（或 `doc`）变量 → 直接选上，成。
- 如果 `files` 只能填文字、选不了文件变量 → 说明踩了那个坑，改用下面的稳做法。

**稳做法（用 HTTP 请求节点，强烈推荐）**：加一个 **HTTP 请求** 节点：
- 方法：`POST`
- URL：`https://rule-harness-demo.onrender.com/api/dify/upload`
- Body：选 **form-data**，加一个字段 key 填 `files`，**值类型选「文件」**，引用 `sys.files`（或开始节点的 `doc`）。
- ⚠️ 不要手动加 `Content-Type` 头（系统会自动加 multipart 边界，手填反而会坏）。
- 这一步返回的 body 里有 `batch_id`，记下引用名（例如 `{{#http1.body.batch_id#}}`）。

### 节点 3：循环轮询状态，直到完成

加一个 **循环 / 迭代（Loop / Iteration）** 节点，循环体里放：

1. **工具节点 `getBatchStatus`**（或 HTTP GET `.../api/dify/batches/{batch_id}/status`），`batch_id` 填上一步拿到的值。
2. **条件分支（IF/ELSE）**：判断返回的 `status`
   - 等于 `success` 或 `partial` → 跳出循环。
   - 否则 → 接一个 **等待/Wait** 5～10 秒，继续下一轮。
3. 给循环设**最大次数**（例如 60 次 ≈ 10 分钟）兜底，别让它无限转。

### 节点 4：取规则结果

循环结束后，加 **工具节点 `getBatchRules`**（或 HTTP GET `.../rules.json`），`batch_id` 同上。
- 返回的 JSON 里 `rules` 就是抽出来的规则数组。

### 节点 5：用结果（可选）

后面接 **LLM 节点 / 代码节点 / 直接输出**，把 `rules` 拿去总结、过滤、回显给用户，随你。

链路一图：

```
开始(File: doc)
   └─ ② 上传(HTTP/工具) ──► 拿到 batch_id
        └─ ③ 循环 { 查状态 → 没好就等几秒再查 } ──► status=success 跳出
             └─ ④ 取规则 rules.json ──► ⑤ LLM/输出
```

## 第三部分：测试

1. 工作流右上角点 **运行 / 预览**，上传一份小一点的合同（先用几页的，别一上来就传超大文件）。
2. 看每个节点的输出：上传节点要有 `batch_id`；循环里 `status` 会从 `running` 变 `success`；取规则节点要有 `rules` 数组。
3. 全绿就成了。

## 更好 / 更省心的接入方式（建议）

- **上传一律用 HTTP 请求节点**，别纠结自定义工具能不能绑文件——HTTP 节点对文件支持最稳，是 Dify 官方文档的标准做法。自定义工具留给"查状态/取规则"这两个纯 GET 接口最合适。
- **文件别太大**：HTTP 节点传 2MB 以上的文件容易失败。先拿小文件跑通；要传大文件，后续我可以给后端加"传 URL/分片"的接口。
- **MCP 方式**（你截图里有 MCP 标签）：也能接，但同样要处理"异步等待"，而且配置更复杂，对小白不友好，**暂不推荐**，先用上面的方式跑通。
- **稳定性（重要）**：后端目前把批次状态存在内存里，单实例、单进程下没问题；但 Render 免费实例**长时间没人用会休眠**，第一次调用要等几十秒唤醒（轮询会自动等到）。如果以后要正式用，建议我帮你把状态落到数据库 + 给接口加密钥，这两件事做完就很稳了。
