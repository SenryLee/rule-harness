# 规则梳理工具改造方案 v1.2

> 基于对当前代码的完整梳理（classifier.py / preview.py / document_profile.py / parsers.py / p1_body.py + P1 prompt / orchestrator.py / llm.py / config.py / exporter.py / 前端 ConfigDrawer / Settings / WorkbenchView）。
> 四个问题各自给出：根因 → 改造方案 → 验收标准。最后给实施顺序。

---

## 诊断总览

| 问题 | 根因（一句话） | 改造核心 |
|---|---|---|
| 1. 文件匹配不精准 | 三套互不一致的关键词分类器并存，LLM 分类只看前 2000 字，文件名权重过高 | 统一分类服务 + 多点采样 + 置信仲裁 |
| 2. 解析深度不够、规则少 | 三要素是"可留空的附属字段"而非规则骨架；按段落切块导致跨段要素拼不齐；失败块整块丢弃 | 三要素改为必填骨架 + 按"条"切块 + 失败重排队 |
| 3. 速度慢（~3分钟/文件） | 每个自然段一次 LLM 调用（百段=百次调用），全局并发钳死在 8，每次调用携带数千字 system prompt | 块合并（降一个数量级的调用数）+ 连接复用 + prompt 瘦身 |
| 4. 交互冗余、颗粒度无效 | granularity 只有 fine/balanced 两值且只改 prompt 里一句话；4 个入口都能改同一配置 | 颗粒度改 1–5 滑条并挂接到 5 个真实参数；配置入口收敛为"任务级+全局级"两层 |

---

## 问题一：传入文件匹配不精准

### 现状与根因

1. **三套分类器并存、词表互不一致**：
   - `classifier.py` 的 `_GENRE_RULES`（7 体裁 + 反向词）
   - `preview.py` 的 `_SOURCE_RULES` + `_classify_source`（6 来源标签，另一套词表）
   - `document_profile.py` 的 `_DOC_TYPE_RULES`（又一套 document_type）
   三者各自打分、各自给证据，结论可能互相矛盾，前端把它们混在一起展示。改一处词表另两处不会跟着变。
2. **LLM 分类只看前 2000 字**（`classify_with_llm` 的 `text[:2000]`；preview 提取也只取 2048 字）。法规、判决书、手册的前 2000 字常是封面、目录、文号——最没有区分度的部分。
3. **LLM 一票否决预筛**：`merge_results` 中只要 LLM 返回了合法 genre 就直接采纳，预筛阶段辛苦设计的反向信号（anti_filename_kw）全部作废。
4. **文件名权重过高**：filename 命中一词得 `weight*1.5`，正文命中只有 `weight*0.5`。文件名叫"案例汇编"的培训材料会被拖向裁判文书。
5. **行业/合同类型识别**（`_classify_profiles`）是纯词频计分，泛词（验收、服务、担保）虽降权但仍参与，长文档天然占便宜。

### 改造方案

**P1-A 统一分类服务（结构性，优先做）**
- 新建 `backend/classification/`，合并三套词表为一份 `taxonomy.yaml`（体裁 7 类 × 权威 5 级 × 来源标签映射 × 特征标签），`classifier.py`、`preview.py`、`document_profile.py` 全部改为从这一份读取。三个旧模块退化为薄壳，词表只维护一处。

**P1-B 多点采样替代"只看头部"**
- `extract_preview_text` 改为三段采样：头部 1500 字 + 中部 1000 字 + 尾部 500 字（PDF 取第 1 页 + 中间页 + 末页）。判决书的"本院认为"、法规的"附则/施行日期"、手册的章节正文都在中后部，这是当前误判的最大来源。
- LLM 分类的 user prompt 同步给三段采样 + 目录结构（若解析出 heading 块，附前 20 个标题）。标题列表对区分"手册 vs 法规 vs 合同"的区分度远高于正文前 2000 字。

**P1-C 置信仲裁取代"LLM 一票否决"**
- 规则：LLM 与关键词预筛结论一致 → 取两者置信度较高者；不一致且 LLM confidence ≥ 0.8 → 取 LLM；不一致且 LLM confidence < 0.8 → 标记 `needs_confirmation=true`，前端该文件的分类下拉框高亮黄色 + 展示两边证据，让用户一键二选一。**不精准的兜底不是更聪明的算法，而是把分歧亮出来。**
- 文件名/正文权重对调：filename 命中降为 `weight*0.8`，正文命中升为 `weight*1.0`，正文采样范围从 500 字扩到 3000 字。

**P1-D 行业匹配去噪**
- `_classify_profiles` 中泛词（`_GENERIC_PROFILE_TERMS`）直接不计分（现在还计 0.25~1.0）；强关键词要求"≥2 个不同词命中"才达到 auto_apply 阈值，单词命中只给建议不自动应用。

### 验收标准
- 准备 20 个已知类型的标注文件（含易混淆的：法院出的操作指引、名字带"案例"的培训手册、含"红线"字样的普通制度），体裁分类准确率 ≥ 90%，且所有判错的文件必须带 `needs_confirmation` 标记（允许不准，不允许"自信地错"）。

## 问题二：规则解析深度不够、数量太少、三要素不落地

### 现状与根因

1. **三要素是附属字段，不是骨架**。P1 prompt 中 assumption / behavior_mode / consequence 被定义为"规则级可审计分析字段"，且明文写着"没有对应内容时填空字符串"——模型自然大量留空。主交付物 main.csv（7 列）根本不含这三列，三要素只进 metadata.csv。**用户看到的产出里三要素几乎不存在，这是"深度不够"的直接体感来源。**
2. **按自然段/双换行切块，跨段要素拼不齐**。法律规则的假定条件经常在条的首款、行为模式在第二款、法律后果（违反的，处……）在最后一款，甚至在"法律责任"章。每段独立调用 LLM，模型永远看不到完整的三要素，只能输出残缺规则。
3. **规则少的三个泄漏点**：
   - 429 重试耗尽后 `record_llm_failure` 返回 `[]`——**整块静默丢弃**（你之前发现的 429 风暴问题的残留路径仍在）；
   - `informational=true` 一票跳过整段；
   - `validate_atomic` 的硬约束（check_item ≤30 字、含连接词即 fail、双数字即 multi_threshold）让大量合法规则 struct_fail 降置信，再被阈值挡掉。
4. 法规/判决类文本和合同文本共用同一个 P1 prompt，决策树是围绕合同条款设计的（数义务主体、数阈值），对"假定条件—行为模式—法律后果"式的法条结构不友好。

### 改造方案

**P2-A 三要素升级为规则骨架（核心改动）**
- P1 prompt 重写输出结构：每条规则**必须**先输出三要素，再从三要素推导出 check_item / requirement：
  ```json
  {
    "assumption":   "假定条件：何种场景/前提下本规则适用（必填，禁止空串）",
    "behavior_mode":"行为模式：谁、应当/不得/可以、做什么（必填）",
    "consequence":  "法律后果：违反或满足的效力/责任/风险（必填；原文确无后果时写'原文未规定后果，风险为×××'并标记 consequence_source=inferred）",
    "exception_conditions": "例外/豁免（可空）",
    "check_item": "由三要素压缩出的检查项",
    "requirement": "[条款]/[合规] + 行为模式的审查口径"
  }
  ```
- few-shot 全部重写为"先三要素后规则"的示范，特别加一个法条例子（如《民法典》某条 → 拆出假定+行为+后果）。
- 后端 `validate_atomic` 新增校验：assumption / behavior_mode 为空 → struct fail；`consequence` 为空且未标 inferred → 触发**一次定向补全调用**（只把该规则 + 原文段落发回去，要求补三要素），补不出来才放行并标记。
- **main.csv 从 7 列扩为 10 列**：加入 假定条件 / 行为模式 / 法律后果 三列（或按你的交付格式要求保持 7 列主表 + 单独导出 `rules_三要素.csv`，二选一，建议直接扩列）。

**P2-B 按"条"切块（与问题三共用同一个改动）**
- 新增 `backend/chunker.py`：在 parsers 产出段落块之后、进 P1 之前做一次**结构化重组**：
  - 识别 `第X条 / 第X款 / X.X / （一）` 等条文边界，一"条"及其下属款项合并为一个语义块；
  - 非法条文本按标题层级聚合，目标块大小 1200–2500 字（颗粒度滑条可调，见问题四）；
  - 每块携带上下文头：`【所属章节】第三章 法律责任 ＞ 第四十二条`，跨章引用的后果条款可被模型关联。
- 这一改动同时解决"跨段三要素拼不齐"（深度）和"调用次数过多"（速度）。

**P2-C 堵住规则泄漏**
- LLM 失败的块不再静默丢弃：进入 `failed_blocks` 队列，批次主流程跑完后**串行低速重试一轮**，仍失败才记 error 并在 summary 中按块列出（现在只有一行 error 字符串）。
- `informational=true` 时要求模型必须给出 `skip_reason`，导出到 `skipped_blocks.csv` 供抽查——让"漏"变得可审计。
- `validate_atomic` 放宽：check_item 上限 30→40 字；连接词检查只对"且/并且"保留，"和/以及"出现在名词并列（如"设计和施工"）不再判罚（用简单启发：连接词前后都是名词性短语则放行，实现上可先只判罚谓语级连接）。

**P2-D 法规/判决专用提示词**
- 按 document_genre 路由：`法律法规 / 监管与司法文件` → 新 prompt `P1_law_extract.txt`（决策树围绕三要素：一条法条内有几组"假定+行为"就拆几条规则；法律责任章节的条文要回链到对应行为条文）；`裁判文书` 已有 P5；其余走现有 P1。

### 验收标准
- 用一部你熟悉的法规（如 30 条的部门规章）做基准：规则条数 ≥ 条文数 × 1.2；三要素非空率 ≥ 95%；抽 10 条人工核对三要素与原文对应关系。
- summary 中 `low_output_files` 为空或每个文件都能给出可解释的 skip 原因。

## 问题三：处理速度（~3 分钟/文件）

### 现状与根因（按影响排序）

1. **调用次数爆炸**：docx 按自然段、PDF 按双换行切块，一份 50 页合同轻松切出 150+ 块，每块一次 LLM 调用。这是 3 分钟的主因。
2. **全局并发硬钳 8**：`llm.py` 中 `_DEFAULT_MAX_CONCURRENCY=8`（上次 429 风暴后的保守兜底），150 次调用 ÷ 8 并发 × 每次 5–10 秒 ≈ 2–3 分钟，对得上你的体感。
3. **每次调用的固定开销大**：system prompt 含全量 theme_key 白名单 + 6 个 few-shot，约 4000+ 字；且每次调用新建 `aiohttp.ClientSession`（TLS 握手不复用）。
4. rpm_limit 语义混乱的历史问题已用硬钳兜底，但没有真正的"按分钟限速"——并发 8 是拍的，既可能浪费额度也可能仍触发限流。

### 改造方案

**P3-A 块合并（= P2-B，一份代码两个收益）**
- 1200–2500 字/块后，150 块 → 15–25 块，调用数降一个数量级。单块输出规则数变多（一次输出 5–15 条），总时延预期从 ~3 分钟降到 **30–60 秒/文件**。
- max_tokens 需同步从 4000 上调（建议 8000，DeepSeek/MiMo 都支持），防止多规则输出被截断。

**P3-B 真令牌桶替代信号量钳制**
- `LLMProvider` 实现按 rpm_limit 的滑动窗口限速（每分钟最多 N 个请求**开始**）+ 并发上限两层；429 时全局退避（一个请求收到 429，整个 provider 暂停 retry_after 秒，而不是各请求各自重试——避免重试风暴）。`LLM_MAX_CONCURRENCY` 默认提到 16，块变大后单请求时间变长，16 并发不会超 DashScope 额度。

**P3-C 固定开销削减**
- `aiohttp.ClientSession` 升为 provider 级单例（在 `__init__` 创建，批次结束 close）。
- theme_keys 白名单从"全量平铺"改为两级：system 里只给一级主题（约 20 个），让模型输出 `payment.late_fee.floor_rate` 式的具体 key 时**不再强制白名单匹配**，改为后处理就近映射（编辑距离/前缀匹配到白名单，映射不上的进 `theme_key_unmapped` 标记而非 struct fail）。system prompt 可减约 1/3。
- 固定不变的 system 前缀放最前（DeepSeek 自动 context caching 按前缀命中，可省 token 费与首 token 延迟）——把 `{document_profile}`、`{task_guidance}` 等动态段移到 user 消息。

**P3-D 进度可感知（配合问题四）**
- 块合并后块数变少，进度条粒度变粗；`mark_pipeline_block_done` 改为按"预计 token 完成量"上报，前端进度平滑。

### 验收标准
- 基准：同一份 50 页 docx，改造前后各跑 3 次。目标 P50 ≤ 60 秒/文件；429 次数为 0；规则数不低于改造前（块合并不应漏规则，由 P2-C 的 skipped_blocks.csv 验证）。

## 问题四：交互冗余、颗粒度配置没发挥作用

### 现状与根因

1. **颗粒度本质是个假开关**：`granularity` 只有 `fine|balanced` 两个值，UI 是 radio（不是你要的拉条），且唯一作用是替换 prompt 里一句话（`_coverage_policy`）。模型对这种措辞级差异不敏感，所以你感知不到任何效果。
2. **同一配置 4 个入口**：ConfigDrawer、SettingsView、Settings 页、WorkbenchView 都能改 extraction 配置，互相不同步，用户搞不清在哪改才算数；且全部是全局配置，没有"本次任务用什么颗粒度"的概念。
3. 置信度权重、优先级数字等高级配置直接暴露给法务用户，认知负担大但实际从不调整。

### 改造方案

**P4-A 颗粒度改为 1–5 滑条，并挂接到真实参数**

新增 `extraction.granularity_level: int (1–5)`，旧 fine/balanced 映射为 4/3 兼容。每一档同时驱动五个真实行为：

| 档位 | 目标块大小 | prompt 拆分策略 | informational 门槛 | dedupe 合并阈值 | 每千字规则数提示 |
|---|---|---|---|---|---|
| 1 粗 | 3000 字 | 只取强义务条款，RuleCount=min(N,K) | 宽松跳过 | 激进合并 | 0.5–1 条 |
| 2 | 2500 | 合并同主体同主题 | ↑ | ↑ | 1–2 |
| 3 平衡(默认) | 2000 | 现行 balanced 策略 | 现行 | 现行 | 2–4 |
| 4 细 | 1500 | 现行 fine 策略 | 收紧（必须给 skip_reason） | 保守合并 | 4–6 |
| 5 极细 | 1200 | RuleCount=max(N,M,K,D)，例外条件单独成规则 | 几乎不许跳过 | 仅完全同指纹合并 | 6–10 |

关键点：**"每千字规则数提示"写进 prompt，且后处理做数值校验**——某块输出规则密度低于档位下限时，该块标记 `under_extracted` 进入复查队列（颗粒度终于有了可被验证的效果，而不是一句措辞）。

**P4-B 配置收敛为两层**
- **任务级**（创建任务时设置，随任务保存）：颗粒度滑条、任务模式、范围说明、我方立场、本次行业方案。WorkbenchView/TaskNew 只留这些。`run_batch` 的 `file_metas`/任务参数中加 `granularity_level`，覆盖全局默认。
- **全局级**（Settings 单一入口）：模型/API、并发、预算、OCR、存储、置信度权重（折叠进"高级"）。删除 ConfigDrawer 与 SettingsView 的重复实现（二者保留一个），Settings 页与 WorkbenchView 不再出现全局配置项。
- 优先级数字、置信度权重默认折叠，顶部加"恢复默认"。

**P4-C 让已有但无感的功能浮出水面**
- 分类证据（evidence）、needs_confirmation（P1-C）、skipped_blocks / under_extracted（P2-C/P4-A）都要在结果页有入口，否则又是"做了但没发挥效果"。
- 结果页规则卡片直接展示三要素（assumption/behavior_mode/consequence 三行），与 main.csv 扩列对齐。

### 验收标准
- 同一文件分别用档位 1/3/5 跑：规则数应呈现明显单调递增（如 15 / 40 / 80 条量级），且档位 5 的规则平均长度更短（更原子）。
- 新用户路径：创建任务页只需做 3 个决定（传文件、拉颗粒度、选模式）。

## 实施顺序（建议分四批，每批可独立上线验证）

| 批次 | 内容 | 预期收益 | 依赖 |
|---|---|---|---|
| ① | P2-B/P3-A 块合并 + P3-B 令牌桶 + P3-C 连接复用 | 速度 3min→1min 内；429 清零 | 无 |
| ② | P2-A 三要素骨架 prompt + 校验补全 + main.csv 扩列 + P2-C 泄漏修补 | 深度与数量的体感质变 | ①（块大了三要素才拼得齐） |
| ③ | P4-A 颗粒度滑条五参数联动 + P4-B 配置收敛 | 颗粒度真实可感 | ② |
| ④ | P1-A/B/C/D 统一分类 + 多点采样 + 仲裁 | 匹配准确率 ≥90% | 无（可与①并行） |

每批的验收标准如上各节；建议在 `backend/tests/` 为 ①② 各加基准回归测试（固定样本文件 + 规则数下限断言），防止后续改 prompt 时回退。

## 风险提示

- 块合并后单次输出规则变多，JSON 截断风险上升 → max_tokens 上调 + `chat_json` 加截断检测（解析失败且 finish_reason=length 时对该块二分重切）。
- 三要素必填会拉低单条规则生成速度（输出 token 变多），被 ① 的调用数下降覆盖，净收益仍为正。
- main.csv 扩列若有下游系统（Dify 工作流）按 7 列解析，需同步更新（你之前的 Dify 接入按"胖后端瘦工作流"模式，改动应集中在后端导出层，工作流侧只是透传，影响可控）。
